# [OBSOLETE] WiFi AP Not Visible After ESP32-S3 Migration — Rust Main() Never Executed

**Status: SUPERSEDED (2026-07-08)** — The C++23 rewrite fixes this issue.
The firmware now boots fully, creates a SoftAP "EcoTiter-FCD2", runs an
HTTP server, and serves a captive portal. See the migration plan for
current status.

## Original Problem (Rust firmware)

After ESP32→ESP32-S3 migration (commits 90cc950, 886b727), the WiFi AP
"EcoTiter-AP" is not visible on any client scan. The firmware boots
correctly through the ESP-IDF C startup (PSRAM init, heap init), and the
log shows `main_task: Calling app_main()`. However, **no Rust output
appears after that line** — not even a boot marker placed as the very
first statement in `fn main()` before any initialisation.

## Investigation History

### Hypothesis 1: Duplicate `app_main` symbol (2026-07-07)

**Theory:** The CMake-generated `target/.../out/main/main.c` contains an
empty `void app_main() {}` stub. When the linker resolves `app_main`, it
picks the empty C definition instead of the Rust definition from
`esp-idf-sys/src/start.rs`.

**Evidence for (original):**
- `nm` on final ELF: `app_main` = 16 bytes, `main` = 22 bytes — both tiny
- Boot marker via MMIO UART at `ecotiter::main()` entry — **not printed**
- `xtensa-esp-elf-objdump -d` of `app_main` — bytes do not decode to valid Xtensa mnemonics

**Verdict: REJECTED.** Linker map analysis (`libespidf.map`) conclusively
shows `app_main` resolves from the Rust rlib, not the C stub:

```
.text.app_main  0x420e7c98  0x10
  .../libesp_idf_sys-*.rlib(esp_idf_sys-*-cgu.1.rcgu.o)
  0x420e7c98  app_main
```

The CMake stub (5 bytes, `entry a1, 32; retw.n`) is **not** selected.

### Hypothesis 2: LLVM Xtensa backend generates invalid instructions (2026-07-07)

**Theory:** The `esp` Rust toolchain's forked LLVM has a buggy Xtensa LX7
backend that produces instruction encodings the ESP32-S3 cannot execute.

**Evidence for (original):**
- `xtensa-esp-elf-objdump` (GNU binutils) fails to decode ALL Rust
  functions — shows only raw bytes, no Xtensa mnemonics
- C functions (compiled by `xtensa-esp-elf-gcc`) decode correctly

**Verdict: REJECTED.** The GNU disassembler simply cannot decode LLVM's
Xtensa instruction encodings. The official `llvm-objdump` from the
esp-clang toolchain decodes the same bytes correctly:

```
# llvm-objdump (esp-clang 20.1.1):
420e8cf4: 36 41 00     entry  a1, 32
420e8cf7: b1 c0 9f     l32r   a11, . 32513
420e8cfa: 81 c0 9f     l32r   a8, . 32514
420e8cfd: 0c 0a        movi.n a10, 0
420e8cff: e0 08 00     callx8 a8
420e8d02: 1d f0        retw.n

# xtensa-esp-elf-objdump (GNU binutils):
420e8cf4: b1004136          (undecoded)
```

The Rust code contains standard Xtensa instructions (`entry`, `l32r`,
`movi.n`, `callx8`, `retw.n`, `mov.n`, `s32i.n`, etc.) that the CPU
will execute correctly. Confirmed across 4 esp toolchain versions
(v1.94.0.2, v1.95.0.0, v1.91.1.0, v1.96.0.0) — all produce the same
valid instructions.

### Current Status

**Root cause: UNKNOWN.** Both hypotheses are ruled out. The symtom
persists: `app_main()` is called by FreeRTOS, the Rust `app_main` IS
present in the binary, yet `ecotiter::main()` never executes (boot marker
not printed).

Possible directions for further investigation:

1. **Stack corruption on entry:** `app_main()` in `start.rs` calls
   `main(0, &[null])` — the autogenerated C main bridge. The Rust runtime
   init (`lang_start_internal`) may crash before reaching the user `main()`
   due to stack misalignment or insufficient stack.
2. **PSRAM + Rust allocator interaction:** PSRAM is initialised by the C
   startup, but Rust's global allocator may be configured incorrectly for
   the S3's PSRAM mapping.
3. **Vector table / exception handling:** The Rust panic handler or
   exception handling setup may differ between LX6 and LX7.
4. **Hardware breakpoint/wdt:** The WDT is disabled early in the C startup,
   but the Rust code path might trigger a different hardware watchdog.

## Toolchain notes

- Recommended esp toolchain: **v1.96.0.0** (`rustup toolchain install esp --force` reinstalls latest)
- Always use `llvm-objdump` (not GNU `xtensa-esp-elf-objdump`) to disassemble Rust code:

  ```bash
  /home/vlabe/.espressif/tools/esp-clang/esp-20.1.1_20250829/esp-clang/bin/llvm-objdump \
    -d target/xtensa-esp32s3-espidf/debug/ecotiter
  ```

- The GNU binutils disassembler cannot decode LLVM-generated Xtensa instructions;
  this is a known incompatibility, not a bug in either tool.

## Edge cases

1. **Without PSRAM** (`CONFIG_SPIRAM=n`): Rust `main()` also fails to
   execute — the issue is in the Rust entry-point chain, not PSRAM.
2. **Debug vs Release build**: The same symptom appears in both profiles.
3. **Different USB-UART vs USB-Serial/JTAG**: The MMIO UART boot marker
   bypasses all VFS/stdout layers, so the issue is not a console-output
   routing problem.
4. **`SPIRAM_USE_MALLOC`**: Toggling this option does not affect the
   symptom — PSRAM initialisation completes successfully before `app_main()`.
