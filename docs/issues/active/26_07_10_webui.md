# WebUI Dashboard and WebSocket Broadcast Issues

## 1. WebSocket frames do not reach frontend

### Problem
Dashboard fields (temperature, ADC, valve, burette) show `"--"` indefinitely. The WebSocket connection is established (`ws.readyState === 1`, `connectionAlive === true`), but `onmessage` never fires for broadcast data or log events.

### Root cause
`broadcastWsEvent()` in `http_server.cpp` had several issues:
- Diagnostics used `write(STDOUT_FILENO, ...)` which was invisible after UART reinit (see Issue 2)
- No null/empty checks for frame data before sending
- Session invalidation on transient `httpd_ws_send_frame_async` errors was too aggressive
- No `ESP_LOGI`/`ESP_LOGW` diagnostics (re-entrancy-safe)

### Solution
- Replaced `write(STDOUT_FILENO)` diagnostics with `ESP_LOGI`/`ESP_LOGW` (with re-entrancy guard)
- Added null/empty checks for frame data
- Improved session invalidation with logging of fd info state
- Removed session invalidation on transient `httpd_ws_send_frame_async` errors

### Edge cases
- Session fd may become stale if client reconnects (e.g., after flash). `broadcastWsEvent` cleans stale sessions by checking `httpd_ws_get_fd_info()`.
- Multiple WS clients (max 4 sessions). Not tested.

## 2. ESP_LOG forwarding to UART unstable after serial driver reinit

### Problem
After `esp_log_set_vprintf(logVprintf)` is called at the start of `app_main()`, subsequent `write(STDOUT_FILENO, buf, n)` calls in `logVprintf` do not produce visible output in the serial monitor. `puts("DBG: ...")` lines still appear. Broadcast JSON from `serial.write()` also works. Only the forwarded ESP_LOG output is lost.

### Root cause
The `SerialReader::init()` (step 4) calls `uart_driver_install()` for the console UART. This reinitializes the UART driver, which may change the internal file descriptor mapping for `stdout`/`STDOUT_FILENO`. After reinit, `write(1, ...)` may go to a different fd than `puts()`/`fwrite()`.

### Solution
- `logVprintf()` already uses `fwrite(buf, 1, n, stdout)` + `fflush(stdout)` — confirmed working
- All `write(STDOUT_FILENO)` calls removed from `broadcastWsEvent()` (replaced with `ESP_LOGI`/`ESP_LOGW`)

### Edge cases
- Early ESP_LOG calls (before step 4 serial init) may use different UART state than later calls
- `esp_log_set_vprintf` affects ALL ESP_LOG calls — any re-entrancy (ESP_LOG called from within vprintf) is silently dropped by `pushing_` guard

## 3. `/api/logs` response buffer size marginal

### Problem
`ResponseBuffer` (2048 bytes) holds approximately 20 log entries. With longer messages (e.g., JSON payloads, stack traces), fewer entries fit. The response is silently truncated when the buffer overflows, producing invalid JSON.

### Root cause
The `append()` lambda in `api_logs_handler` stops writing when `pos >= rsp.size()`. The JSON closing braces are then written at the truncated position, resulting in unterminated strings.

### Solution
- Increased from 512 to 2048 bytes (covers 20 typical entries)
- Future: implement chunked response or dynamic buffer for large queries

## 4. Frontend render race with WebSocket connection

### Problem
`loadInitialLogs()` (HTTP fetch of `/api/logs`) and `connectWs()` (WebSocket) are both called from `initApp` on `DOMContentLoaded`. If `loadInitialLogs` completes before WS is ready, initial log entries render but subsequent WS-pushed log events may be missed during the reconnect gap.

### Root cause
No synchronization between HTTP fetch and WS connection. The frontend uses `addLogEntry(msg)` for both paths independently, but the WS might not yet be connected when the initial fetch completes.

### Solution
Not critical — messages are pushed to the same `APP_STATE.logs.messages` array regardless of source. The gap only affects the first ~1 second of operation.

## 9. `/api/logs/download` connection reset (found during curl testing)

### Problem
`GET /api/logs/download` caused `Connection reset by peer` after ~7 seconds.

### Root cause
`LogEntry entries[100]` (100 × 140 bytes = 14 KB) allocated on the stack of the HTTP handler. The HTTP server runs in the `net_owner` task with 16 KB stack. A 14 KB array left only 2 KB for function calls and other locals, causing a stack overflow.

### Solution
- Reduced array to `LogEntry entries[20]` (2.8 KB)
- Fixed `snprintf` return value handling: `snprintf` returns the would-be length, not actual bytes written. When the buffer was nearly full, `pos` was incremented past the buffer boundary, causing `httpd_resp_send()` to read past the buffer. Fixed by using `std::min(n, remaining - 1)` for actual written bytes.
